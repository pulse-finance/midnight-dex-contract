The review focused on on-chain guarantees in the Compact source: asset custody,
privileged roles, AMM arithmetic, order lifecycle liveness, callback safety, and
edge-case state transitions. Tests were used only as auxiliary evidence and not
as the specification; behaviors covered by tests are still reported when the
contract invariant appears unsafe.

Midnight standard-library primitives such as `receiveShielded`, `sendShielded`,
`mergeCoin`, `mintShieldedToken`, `kernel.claimContractCall`,
`persistentHash`, and `transientCommit` were treated as trusted.

## Findings

### H-01: AMM initialization is unpermissioned and zero-supply reinitialization is underconstrained

Severity: **High**

Affected code:

- `src/Amm.compact:93-133`
- `src/Amm.compact:99`
- `src/Amm.compact:102`
- `src/Amm.compact:1160-1167`
- `src/Amm.compact:1203-1205`

`AmmInitXYLiq` is callable by anyone and only checks
`lpCirculatingSupply == 0`. The successful caller writes
`batcherCommitment`, receives the initial X/Y coins, mints LP, and thereby
controls every `assertOnlyBatcher` path, including order processing,
`AmmUpdate`, treasury changes, and reward collection.

The same weak guard also permits later reinitialization whenever LP supply
returns to zero. The circuit does not check that `active` is empty, that
`xLiquidity`, `yLiquidity`, and `xRewards` are zero, or that reserve coin
positions and order slots are empty. This can reset accounting and batcher
authority over non-empty or terminal pool state.

Impact:

- A pool not initialized atomically by trusted deployment infrastructure can be
  seized by the first initializer.
- A zero-supply pool can be reinitialized over dust, unpaid state, or even an
  active order.
- Fee and treasury authority are ultimately tied to the initializer's witness,
  not to a constructor-defined admin or immutable deployment parameter.

Recommendation:

- Make batcher authority explicit in the constructor, or require
  `AmmInitXYLiq` to prove an expected deployment/admin commitment.
- Add a one-time `initialized` flag, or a separate retired state, rather than
  using `lpCirculatingSupply == 0` as the only initialization guard.
- If reinitialization is intentionally unsupported, require `active.is_none`,
  zero liquidity/rewards, empty reserve coin positions, and no live order state
  before initialization can proceed.

### M-01: Post-open order funds have no user-controlled recovery path if the batcher stalls

Severity: **Medium**

Affected code:

- `src/MarketOrder.compact:60-115`
- `src/MarketOrder.compact:156-183`
- `src/MintLpOrder.compact:69-153`
- `src/MintLpOrder.compact:189-246`
- `src/BurnLpOrder.compact:62-118`
- `src/BurnLpOrder.compact:173-247`

The order contracts allow owner cancellation only in the initial `Opened` state.
After an AMM slot is reserved, users cannot cancel even if their coin is still
held by the order contract. After a coin is sent to the AMM, every completion
path depends on the batcher continuing the AMM lifecycle and paying back through
the callback.

Examples:

- `MarketOrderCancel` only accepts `Opened`; `Reserved` and `Sent` have no user
  refund path.
- `MintLpOrder` can reach `SentX` with X sent to the AMM and Y still held by the
  order contract, but there is no owner recovery path from `Reserved` or
  `SentX`.
- `BurnLpOrder` can only cancel while `Opened`; once reserved or sent, LP
  recovery depends on AMM completion.

Impact:

- A stalled, malicious, or unavailable batcher can lock user funds without an
  on-chain timeout or user reclaim mechanism.
- This is a custody liveness issue even when the batcher cannot directly steal
  through the contract arithmetic.

Recommendation:

- Add timeout/cancel circuits for `Reserved` and partially sent states.
- For AMM-sent states, consider a protocol-level refund/expiry path that either
  cancels the AMM order before activation or returns unprocessed order coins.
- If the batcher is intentionally fully trusted for liveness, document that
  trust requirement prominently in user- and integrator-facing docs.

### M-02: Inclusive rounding bounds allow one-unit underpayment or fee overcharge on exact divisions

Severity: **Medium**

Affected code:

- `src/Amm.compact:407-420`
- `src/Amm.compact:459-460`
- `src/Amm.compact:525-526`
- `src/Amm.compact:590-591`
- `src/Amm.compact:601-602`
- `src/Amm.compact:634-635`
- `src/Amm.compact:647-648`
- `src/Amm.compact:679-691`
- `src/Amm.compact:732-744`
- `src/Amm.compact:749-750`
- `src/Amm.compact:798-816`

Many AMM validations implement floor/ceil checks with inclusive upper and lower
bounds. This admits an adjacent worse value whenever the exact division has no
remainder.

For output calculations, the constant-product checks accept both the correct
post-swap reserve and a reserve one unit larger when the division is exact. That
means the user can receive one unit less than the mathematically exact output.

For fee calculations, the ceil checks accept a fee one unit higher when the fee
division is exact. For example, with a 10 bps fee and `xIn = 1000`, the exact
fee is `1`, but the current bounds also accept `xFee = 2`.

The same pattern appears in LP minting and withdrawal calculations. For exact
proportional deposits or withdrawals, the validator can accept one fewer LP
token or one fewer withdrawn token than the exact formula requires.

Impact:

- The batcher can systematically round against users by one base unit in exact
  division cases.
- Fee overcharge can accrue to `xRewards` and later be sent to the treasury.
- Underpaid swap, mint, and withdrawal amounts remain in the pool, benefiting
  LPs rather than the executing user.

Recommendation:

- For floor results, enforce `amount * denom <= numerator` and
  `(amount + 1) * denom > numerator`.
- For ceil fees, enforce `(fee - 1) * denom < numerator` and
  `fee * denom >= numerator`, with explicit handling for zero numerator and
  zero-fee pools.
- For constant-product output checks, make the "one less output" side strict,
  e.g. require the lower product to be strictly below the starting `k`.
- Add tests for exact-division cases and assert the expected mathematical
  result, not merely current accepted behavior.

### M-03: Validators can commit states that split circuits cannot finalize

Severity: **Medium**

Affected code:

- `src/Amm.compact:666-695`
- `src/Amm.compact:714-767`
- `src/Amm.compact:781-834`
- `src/Amm.compact:883-908`
- `src/Amm.compact:924-955`
- `src/Amm.compact:969-980`

Withdrawal validators can accept outputs that consume an entire reserve coin,
but `AmmSplitX` and `AmmSplitY` require `sendShielded` to return a change coin.
If validation is committed with `xOut` equal to the full X reserve coin value,
or `yOut` equal to the full Y reserve coin value, the later split fails with
`change.is_some` false.

After validation, the active order state is no longer `Unvalidated`, so the
batcher cannot choose different output arguments. The active order remains
stuck, and `AmmDeactivateOrder` cannot run because the split state was never
reached.

Impact:

- A valid-looking withdrawal can permanently block the AMM active slot.
- Full or terminal withdrawals can turn into unrecoverable liveness failures.
- This interacts badly with H-01 because zero LP supply can also reopen the
  initialization path while old state may still exist.

Recommendation:

- Reject validator outputs that would leave no reserve coin change when the
  follow-up split requires change.
- Or update split circuits to handle exact-full sends by removing the reserve
  coin and still completing the order.
- Add tests where validation succeeds but the following split would currently
  fail, and require the full lifecycle to be finalizable.

### M-04: Order contracts provide no on-chain price, fee, or expiry protection

Severity: **Medium**

Affected code:

- `src/Common.compact:38-44`
- `src/MarketOrder.compact:6-14`
- `src/MintLpOrder.compact:6-15`
- `src/BurnLpOrder.compact:6-14`
- `src/MarketOrder.compact:119-138`
- `src/MintLpOrder.compact:155-174`
- `src/BurnLpOrder.compact:120-150`
- `src/Amm.compact:1160-1167`

The order structs commit to sent amounts, colors, an AMM circuit bundle, and a
return address, but they do not include minimum output, minimum LP minted,
maximum fee, maximum slippage, or expiry/deadline constraints. The receive
circuits accept the amount returned by the AMM callback as long as the local
state permits receiving and the coin matches the configured color.

This is especially important because `AmmUpdate` lets the batcher change the fee
before processing outstanding orders. A user who opened an order under one fee
or pool price has no on-chain guard against later execution under worse terms.

Impact:

- Users rely on off-chain UI and batcher policy for execution protection.
- A valid AMM execution can be economically unacceptable to the user.
- Stale orders can execute after arbitrary pool movement.

Recommendation:

- Add protected order types with min-output/min-LP, max-fee, and expiry fields.
- If the current contracts are intended to be pure market orders, document that
  they provide no on-chain execution protection.
- Consider snapshotting fee assumptions into order data if mutable fees remain.

### L-01: Public return receivers allow post-send coin injection and lifecycle griefing

Severity: **Low**

Affected code:

- `src/MarketOrder.compact:119-138`
- `src/MintLpOrder.compact:155-174`
- `src/BurnLpOrder.compact:120-150`
- `src/MarketOrder.compact:143-154`
- `src/MintLpOrder.compact:176-187`
- `src/BurnLpOrder.compact:155-171`

The AMM return receiver circuits are public. They do not authenticate the caller
as the configured AMM; they accept any shielded coin that satisfies the local
state, amount, nonce, and configured return color checks. `MarketOrder` and
`MintLpOrder` also ignore the return-kind argument.

The AMM payout transaction still needs a matching callback claim for normal
completion, so this is mainly a griefing and accounting-ambiguity issue rather
than direct theft. A third party can inject a matching coin after the order is
sent, occupy the primary or secondary coin position, and force merge/cleanup
steps before close.

Impact:

- Off-chain systems cannot assume every accepted return coin came from the AMM.
- Attackers can create additional merge work or delay closure with spam coins.
- Receiver state is public lifecycle surface area.

Recommendation:

- If Compact exposes stronger caller/callback binding in the future, bind
  receive circuits to the expected AMM payout call.
- Until then, keep merge paths robust and make indexers distinguish claimed AMM
  payouts from unsolicited receiver calls.
- Validate `returnKind` in `MarketOrder` and `MintLpOrder` if callers or
  indexers use it semantically.

### I-01: `calcK` only supports reserves that fit into `Uint<124>`

Severity: **Informational**

Affected code:

- `src/Amm.compact:1213-1215`

AMM reserve ledgers are `Uint<128>`, but `calcK` casts both reserves to
`Uint<124>` before multiplying:

```compact
return ((x as Uint<124>) * (y as Uint<124>)) as Uint<248>;
```

This appears motivated by available Compact integer widths, but it means the
effective reserve ceiling for swap/zap validation is lower than the ledger type
suggests.

Impact:

- Very large pools can become unable to validate AMM operations once reserves no
  longer fit the narrower cast.
- Integrators may overestimate the supported reserve range from the public
  ledger types.

Recommendation:

- Document the effective reserve ceiling.
- Add explicit reserve-size guards before reaching the cast boundary.
- Revisit the implementation if Compact adds a wider integer type.

### I-02: `BurnLpOrder.reset` leaves a stale `ammSlot` value

Severity: **Informational**

Affected code:

- `src/BurnLpOrder.compact:269-272`

`MarketOrder` and `MintLpOrder` reset `ammSlot` to zero when clearing the slot,
but `BurnLpOrder.reset` only clears `state` and `order`. After cancellation or
full close, the public ledger can still show the previous AMM slot while the
contract is `Empty`.

Impact:

- This is unlikely to lock funds because state checks gate the lifecycle.
- It can confuse indexers, diagnostics, or off-chain order reconstruction.

Recommendation:

- Set `ammSlot = 0` inside `BurnLpOrder.reset` for consistency with the other
  order contracts.

## Additional Notes

- `AmmSplitX`, `AmmSplitY`, and `AmmDeactivateOrder` are intentionally public.
  They have no external recipient arguments and only advance the active order,
  so this is not currently a theft vector, but it should remain documented.
- The order `Open` circuits are public single-slot entrypoints. This is fine for
  one-contract-per-order deployment, but shared singleton deployments would be
  vulnerable to slot occupation griefing.
- Several low-value or dust-size orders can validly receive zero output or zero
  LP under the current market-order design. That may be acceptable for explicit
  market orders, but it should not be hidden behind tests that only assert the
  current behavior.
