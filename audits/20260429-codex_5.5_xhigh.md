## Findings

### F-01: Batcher initialization is trusted operationally rather than enforced on-chain

Severity: **Informational** \
Resolution status: **Accepted**

Affected code:

- `src/Amm.compact:80-91`
- `src/Amm.compact:586-591`
- `src/Amm.compact:1048-1050`

`AmmInitXYLiq` is the only circuit that initializes `batcherCommitment`, but it is not protected by any deployer, owner, or expected-batcher check. The first successful caller supplies `batcherSecret()` as their witness and writes:

```compact
batcherCommitment = disclose(generateBatcherCommitment());
```

That caller then becomes the only party able to call `assertOnlyBatcher`-protected circuits, including swaps, deposits, withdrawals, fee/treasury updates, and reward collection. The constructor sets fee, treasury, and token colors, but not the batcher commitment.

Per project clarification, the constructor and `AmmInitXYLiq` are executed consecutively in a trusted environment, and only correctly initialized contracts are surfaced in the dapp. Under that operating model, this is not an exploitable product-path takeover issue for supported pools.

Impact:

- The trust assumption is off-chain and operational, not enforced by the contract itself.
- A third party can still deploy or initialize a lookalike pool instance and become its batcher.
- The risk is mainly to direct contract interaction, broader ecosystem discovery, or future listing/indexing paths that are less curated than the current dapp.

Recommendation:

- If you want the contract to be self-defending outside the curated dapp flow, initialize `batcherCommitment` in the constructor from an expected commitment, or require an explicit deployment/admin commitment in `AmmInitXYLiq`.
- If the current trust boundary is intentional, document it clearly in deployment/runbooks and any integrator-facing docs.
- Keep this issue informational unless pool discovery, indexing, or direct user interaction expands beyond trusted deployments.

### F-02: Zero-supply pools are retired operationally rather than on-chain

Severity: **Informational** \
Resolution status: **Accepted**

Affected code:

- `src/Amm.compact:87`
- `src/Amm.compact:90`
- `src/Amm.compact:93-121`
- `src/Amm.compact:883-885`
- `src/Amm.compact:951-953`
- `src/Amm.compact:1013-1016`

`AmmInitXYLiq` only checks `lpCirculatingSupply == 0`. It does not check that:

- `slot` is empty,
- `xLiquidity == 0`,
- `yLiquidity == 0`,
- `xRewards == 0`,
- reserve coin map positions are empty,
- the caller is an authorized initializer.

This matters because normal withdrawal validators can set `lpCirculatingSupply` to zero while leaving non-zero reserves or a pending send step. The test suite already models the dust-preserving full burn pattern. A local simulator check confirmed this sequence:

1. Initialize `1000 X / 1000 Y / 1000 LP`.
2. Withdraw `1000 LP` for `999 X / 999 Y`.
3. Ledger becomes `lpCirculatingSupply = 0`, `xLiquidity = 1`, `yLiquidity = 1`.
4. `AmmInitXYLiq(100, 100, 100, ...)` succeeds and resets liquidity to `100 / 100`.

Per project clarification, `AmmInitXYLiq` is already at the practical size/complexity limit, and the intended operating model is that a pool which reaches zero LP supply but still contains dust is treated as effectively removed. Under that model, this is not a supported lifecycle path for production pools surfaced by the dapp.

Impact:

- Pool retirement is enforced operationally, not by the contract itself.
- A zero-supply dust-state pool can still be reinitialized on-chain and overwrite prior accounting.
- If direct contract interaction, broader indexing, or future discovery/listing paths ignore the retired status, users may interact with a pool instance that ops considers dead.

Recommendation:

- Treat this as an explicit operational invariant and document that zero-supply pools are retired permanently even if the contract remains callable.
- Ensure indexers, deployment metadata, and dapp discovery paths exclude retired pools.
- Keep this issue informational unless the product later needs on-chain-enforced retirement or permissionless pool discovery.

### F-03: Full-withdrawal dust states are handled operationally rather than in-circuit

Severity: **Informational** \
Resolution status: **Accepted**

Affected code:

- `src/Amm.compact:469-481`
- `src/Amm.compact:500-508`
- `src/Amm.compact:555-563`
- `src/Amm.compact:869-885`

`AmmValidateWithdrawXYLiq` permits exact proportional withdrawals, including withdrawing the full X and Y reserves when `lpIn == lpCirculatingSupply`. However, `AmmSendX`, `AmmSendY`, and `AmmReward` require `sendShielded` to return a change coin:

```compact
assert(sendXResult.change.is_some, "Expected some X change");
assert(sendYResult.change.is_some, "Expected some Y change");
```

If validation is committed with an output amount equal to the entire available coin value, the follow-up send circuit fails. The tests cover this as an expected throw, but the validator still accepts the state transition that sets the slot to `SendX` and updates liquidity/supply.

Per project clarification, this is part of the same intended lifecycle as F-02: pools that reach this kind of terminal dust/full-withdrawal state are treated operationally as removed rather than as pools that must remain finalizable entirely through on-chain logic. Under that model, this is not a supported end-user flow for curated pools.

Impact:

- The terminal-state assumption is operational, not enforced by the contract.
- Full reserve withdrawals still cannot be completed as a clean in-contract lifecycle without leaving dust.
- If external tooling or direct contract users assume every validated withdrawal must remain finalizable on-chain, they can mis-handle retired pool instances.

Recommendation:

- Document this explicitly as part of the pool retirement model.
- Ensure indexers, dapp flows, and operational tooling treat these terminal states as retired pools rather than active pools awaiting completion.
- Keep this issue informational unless the product later requires fully on-chain finalization for terminal withdrawals.

### F-04: Current order contracts intentionally omit price-protection constraints

Severity: **Informational** \
Resolution status: **Accepted**

Affected code:

- `src/Common.compact:32-58`
- `src/MarketOrder.compact:24-51`
- `src/MintLpOrder.compact:25-64`
- `src/BurnLpOrder.compact:25-54`
- `src/MarketOrder.compact:106-125`
- `src/MintLpOrder.compact:132-151`
- `src/BurnLpOrder.compact:110-132`

The order structs store sent amounts, target circuit, return address, return colors, and the AMM tick at send time. They do not store user-provided minimum outputs, minimum LP minted, maximum input-to-output slippage, or expiry/max-tick constraints.

The receive circuits accept whatever amount is returned in the expected color and the close circuits return the stored coins to `returnsTo`. The AMM arithmetic constrains pool-consistent execution, but it does not enforce user-specific price protection.

Per project clarification, this is the intended design for the current order set. These contracts are presently acting as market-style order primitives, and a separate `LimitOrder` contract is planned for future flows that require explicit slippage or price bounds.

Impact:

- Current orders rely on product/UI policy and user intent rather than on-chain min/max execution constraints.
- A valid order can execute after pool movement with no built-in user-specific price bound.
- This is a design tradeoff in the current order family, not an implementation defect relative to the intended behavior.

Recommendation:

- Document clearly that the existing order contracts are market-style and do not provide on-chain price protection.
- When the planned `LimitOrder` contract is introduced, keep price-protected execution isolated there rather than retrofitting assumptions into the current order contracts.
- Keep this issue informational unless product requirements change and these contracts are expected to provide user-level execution bounds directly.

### F-05: Zero-fee pools still require a positive X fee in fee-checked paths

Severity: **Low** \
Resolution status: **Fixed**

Affected code:

- `src/Amm.compact:53-54`
- `src/Amm.compact:586-588`
- `src/Amm.compact:662-663`
- `src/Amm.compact:724-725`
- `src/Amm.compact:785-786`
- `src/Amm.compact:825-826`
- `src/Amm.compact:936-937`
- `src/Amm.compact:998-999`

The constructor and `AmmUpdate` allow `feeBps == 0`. The fee validation pattern then computes `(xFee - 1)` before checking the fee bounds. With `feeBps == 0`, the economically correct fee is `xFee == 0`, but that value underflows in expressions such as:

```compact
assert((xFee - 1) * 10000 <= xIn * feeBps, "Fee too high");
```

Using `xFee == 1` passes the zero-fee bounds, so a configured zero-fee pool still requires a one-unit X-side fee in affected paths.

Impact:

- Zero-fee configuration does not behave as zero fee.
- Correct zero-fee executions can fail, or users can be overcharged by one X unit.

Resolution:

- Fixed by special-casing `feeBps == 0` in all fee-checked AMM validation paths and requiring `xFee == 0` for zero-fee pools.
- Added regression tests covering successful zero-fee swaps and rejection of positive `xFee` for zero-fee swap, zap-in, and zap-out validators.

### F-06: AMM return receivers remain public, but sent-state checks are now enforced

Severity: Low
Resolution status: Partially fixed

Affected code:

- `src/MarketOrder.compact:106-125`
- `src/MintLpOrder.compact:132-151`
- `src/BurnLpOrder.compact:110-132`

Originally, the `ReceiveFromAmm` circuits only required that `slot` was occupied. They did not require `tickUponSend.is_some`.

That has now been tightened:

- all three order contracts require `order.tickUponSend.is_some` before accepting AMM-return coins,
- `BurnLpOrderReceiveFromAmm` continues to use `returnKind` semantically for X/Y routing through `returnColor(...)`,
- `MarketOrderReceiveFromAmm` and `MintLpOrderReceiveFromAmm` still accept a `returnKind` argument only to keep the receive interfaces uniform across order contracts; they rely on the configured `colorReturned` value rather than on `returnKind`.

The remaining issue is that these receive circuits are still publicly callable. In current Compact, there is no tighter way to bind the callback to the AMM sender or to make the entrypoint non-public, so any caller can still submit a coin that satisfies the order's current receive checks after `SendToAmm`.

Impact:

- Pre-send state confusion is now prevented.
- Public post-send coin injection is still possible if a caller can provide a coin matching the accepted return constraints.
- Off-chain indexers and order lifecycle logic still cannot assume that every accepted return coin necessarily came from the intended AMM callback.

Resolution:

- Fixed in part by requiring the order to have been sent before any receive is accepted.
- Added negative unit tests covering both new guard classes.
- The remaining public-access aspect is accepted because Compact does not currently expose a way to couple the receive entrypoint more tightly to the AMM callback origin.
- Interface uniformity is preserved by keeping the `returnKind` argument on all receive circuits even though only `BurnLpOrder` uses it semantically.

### F-07: `calcK` reserve width is constrained by Compact compiler integer limits

Severity: **Informational** \
Resolution status: **Accepted**

Affected code:

- `src/Amm.compact:1052-1053`

AMM reserve ledgers are `Uint<128>`, but `calcK` casts both operands to `Uint<124>` before multiplication:

```compact
return ((x as Uint<124>) * (y as Uint<124>)) as Uint<248>;
```

Initial and per-order amounts are `Uint<64>`, so normal pools are unlikely to hit this quickly. Still, the ledger type permits larger accumulated reserves than `Uint<124>`.

Per project clarification, this is constrained by the current Compact compiler: values wider than `Uint<248>` are not available, so a full-width `Uint<128> * Uint<128>` product cannot be represented directly in-contract. Under that limitation, the narrower cast is a platform constraint rather than an avoidable local implementation choice.

Impact:

- Very large pools can become unable to validate swaps/zaps.
- The effective maximum supported reserve size is lower than the ledger type alone suggests.

Recommendation:

- Document the effective reserve-size ceiling imposed by the compiler/runtime arithmetic model.
- Revisit this if a future Compact release supports a wider integer type for full-width constant-product checks.
- Keep this issue informational unless product requirements approach the reduced reserve bound in practice.

## Additional Notes

- `MintLpOrderOpen` uses one nonce for both deposited coins (`src/MintLpOrder.compact:34-54`). If X and Y colors can ever be equal, use separate nonces and assert distinct asset colors where appropriate.
- The order contracts appear designed for one order per deployed contract. If any deployment uses a shared singleton order contract, public `Open` circuits allow the first opener to occupy the slot until they close it.
